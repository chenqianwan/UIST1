# 合同星座 (Contract Constellation) 上游工作流设计方案

本方案描述了如何将原始合同文本转化为下游「合同星座」系统可消费的结构化节点数据。设计遵循 **“关注点分离”** 原则，强调 **“证据锚定 (Evidence-Grounded)”** 与 **“人类可控 (Human-in-the-Loop)”**，符合 UIST 等 HCI 顶会对智能交互系统的审美。

---

## 1. 设计原则 (Design Principles)

1.  **Nodes-Only Output**：上游仅输出节点列表（Nodes），连线（Links）由下游根据节点的 `parentId` 和 `references` 属性自动派生。
2.  **Evidence-Grounded**：AI 的每一个风险判定和修改建议必须绑定原文证据（Evidence Spans），确保可解释性。
3.  **Policy-Driven Actions**：建议的生成受预设策略约束（如：有删除建议时屏蔽修订建议），而非纯 LLM 自由发挥。

---

## 2. 核心工作流：全量语义合成 (Single-Pass Semantic Synthesis)

*   **任务**：在单一推理周期内，通过一个高度结构化的 System Prompt，将原始合同文本直接映射为下游图谱节点。
*   **具体 LLM 执行步骤 (P)**：
    1.  **全量上下文注入**：将合同全文、法律风险判定准则（Risk Taxonomy）及严格的 JSON Schema 注入模型。
    2.  **锚点预扫描**：要求模型先在内部建立条款索引（锚点），确保不遗漏任何原文段落。
    3.  **无损提取 (Lossless Constraint)**：强制要求 `content` 字段必须是原文的逐字复制，严禁任何形式的简化、概括或精简。
    4.  **层级路径锚定**：要求模型在输出每个节点时，带上显式的层级路径（如 `第一章 > 第一条 > 第(2)款`），以强迫模型保持逻辑一致，确定 `parentId`。
    5.  **风险评估与证据锚定**：同步判定 `riskLevel` (none/low/medium/high) 与 `riskReason`。若风险非 none，必须从原文中摘录出精确的短语作为 `evidence_text`。
    6.  **建议生成与策略过滤**：针对风险点生成 `actions[]` (delete/revise/add_clause)。执行**优先级策略**：若判定为 `delete`，则不生成该节点的 `revise` 或 `add_clause` 建议。
    7.  **TimePhase 推断**：根据内容（如“签约”、“支付”）自动映射至 `pre_sign | execution | ...` 等时间阶段。
    8.  **置信度评估**：计算每条建议的 `confidence` (0.0-1.0)，低置信度高风险操作标记为“需人工复核”。

---

## 3. 核心 System Prompt 完整实现

这是直接给到 GPT-5/Gemini-3 的指令集，确保输出符合 95% 稳定性的要求：

```markdown
# Role
你是一个资深法律科技专家与合同审查员。你的任务是将输入的非结构化合同文本转化为高度结构化的 JSON 图谱数据。

# Core Constraints (不可违背)
1. **LOSSLESS CONTENT**: `content` 字段必须是原文的 100% 逐字复制。严禁概括、精简或修正原文中的任何字符（包括错别字）。
2. **HIERARCHICAL INTEGRITY**: 严格根据条款编号（如 1.1, (1), 第一条）确定父子关系（parentId）。
3. **EVIDENCE GROUNDING**: 所有的风险判定必须附带 `evidence_text`，且该文本必须能从 `content` 中通过字符串匹配找到。
4. **ACTION EXCLUSIVITY**: 如果建议类型为 `delete`，则该节点不允许出现 `revise` 或 `add_clause` 建议。
5. **FIELD ALIGNMENT**: 所有输出字段必须严格匹配下方定义的枚举值。

# Value Definitions
- **type (NodeKind)**:
  - `main`: 合同的核心章节或独立条款（如：付款方式、保密义务）。
  - `sub`: 条款下的具体细则、补充说明、参数或例外情况（挂在对应的 main 节点下，支持嵌套）。
- **riskLevel**: `none | low | medium | high`
- **timePhase**:
  - `pre_sign`: 签约前准备
  - `effective`: 合同生效/背景
  - `execution`: 合同执行/履行
  - `acceptance`: 验收/交付
  - `termination`: 合同终止/解约
  - `post_termination`: 终止后义务/保修
- **action type**: `delete | revise | add_clause`

# Task Steps
1. **Pre-scan**: 扫描全文，识别所有条款的边界、编号和标题。
2. **Structuring**: 构建层级树（root -> main -> sub）。
3. **Risk Analysis**: 针对每个条款，识别法律风险（如：责任不平衡、期限模糊、缺失关键保护）。
4. **Action Generation**: 为有风险的条款生成具体的修改建议。修订建议（revise）必须提供完整的替换文案。
5. **JSON Synthesis**: 按下方 Schema 输出最终结果。

# Output Schema (Strict JSON)
{
  "nodes": [
    {
      "id": "string (使用条款编号，如 'clause_1_1')",
      "label": "string (4-6字标题)",
      "content": "string (原文逐字复制)",
      "type": "main | sub",
      "parentId": "string (父节点ID，根节点指向 'root')",
      "references": ["string (引用的其他条款ID)"],
      "riskLevel": "none | low | medium | high",
      "riskReason": "string (风险解释)",
      "evidenceSpans": [
        { "text": "string (原文中的精确证据短语)" }
      ],
      "timePhase": "pre_sign | effective | execution | acceptance | termination | post_termination",
      "actions": [
        {
          "id": "string (唯一ID)",
          "type": "delete | revise | add_clause",
          "reason": "string (修改理由)",
          "suggestionText": "string (仅在type为revise时提供，替换后的全文内容)",
          "supplementDraft": "string (仅在type为add_clause时提供，补充条款文案)",
          "confidence": number (0.0-1.0)
        }
      ]
    }
  ]
}

# Input Contract Text
[在此处输入合同全文]
```

---

## 4. 防御机制与后处理 (Post-Processing)

由于 LLM 仍有 5% 的幻觉概率，系统需在拿到 JSON 后执行以下自动化校验：

1.  **字符级完整性校验**：
    `if (all_nodes_content.length < original_text.length * 0.98) { trigger_retry(); }`
2.  **证据坐标自动映射**：
    程序自动运行 `node.content.indexOf(evidence_text)` 来计算前端所需的 `start` 和 `end` 偏移量。
3.  **引用合法性过滤**：
    `node.references = node.references.filter(refId => all_node_ids.includes(refId));`

```json
{
  "nodes": [
    {
      "id": "clause_5_2",
      "label": "违约责任",
      "content": "如乙方逾期交付，每逾期一日应支付合同总价 0.1% 的违约金...",
      "type": "sub",
      "parentId": "clause_5",
      "references": ["clause_3_1"],
      "riskLevel": "medium",
      "riskReason": "违约金比例偏低，可能无法覆盖实际损失。",
      "evidenceSpans": [
        { "text": "0.1% 的违约金" }
      ],
      "timePhase": "execution",
      "actions": [
        {
          "id": "act_001",
          "type": "revise",
          "status": "pending",
          "reason": "将违约金比例提升至行业标准 0.5%",
          "suggestionText": "如乙方逾期交付，每逾期一日应支付合同总价 0.5% 的违约金...",
          "confidence": 0.85
        }
      ]
    }
  ]
}
```
