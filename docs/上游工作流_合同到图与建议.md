# 合同星座 (Contract Constellation) 上游工作流设计方案

本方案描述了如何将原始合同文本转化为下游「合同星座」系统可消费的结构化节点数据。设计遵循 **“关注点分离”** 原则，强调 **“证据锚定 (Evidence-Grounded)”** 与 **“人类可控 (Human-in-the-Loop)”**，符合 UIST 等 HCI 顶会对智能交互系统的审美。

---

## 1. 设计原则 (Design Principles)

1.  **Nodes-Only Output**：上游仅输出节点列表（Nodes），连线（Links）由下游根据节点的 `parentId` 和 `references` 属性自动派生。
2.  **Evidence-Grounded**：AI 的每一个风险判定和修改建议必须绑定原文证据（Evidence Spans），确保可解释性。
3.  **Policy-Driven Actions**：建议的生成受预设策略约束（如：有删除建议时屏蔽修订建议），而非纯 LLM 自由发挥。

---

## 2. 核心工作流：三阶段语义流水线 (Three-Pass Pipeline)

*   **任务**：采用“三步走”策略，先稳定生成结构，再产出推理与动作意图，最后由程序执行器落地变更，降低一次性全量推理的幻觉与结构漂移风险。
*   **Stage A — 结构抽取 (Structure Pass)**：
    1.  **全量上下文注入**：输入合同全文与结构化 Schema，仅执行条款边界识别、层级锚定与字段抽取。
    2.  **锚点预扫描**：先建立条款索引（章节/条/款编号与标题），确保不遗漏原文段落。
    3.  **无损提取 (Lossless Constraint)**：`content` 必须逐字复制原文，严禁概括、精简或改写。
    4.  **层级锚定**：确定 `parentId`，仅产出结构字段（`id/label/content/type/parentId/timePhase`）。
    5.  **结构校验门 (Gate)**：通过覆盖率、ID 唯一性、父子一致性校验后，才进入 Stage B。
*   **Stage B — 推理与动作意图 (Reasoning Pass)**：
    1.  **输入固定结构**：读取 Stage A 的节点结构与原文全文，不允许改写结构字段。
    2.  **关系与风险推理**：补全 `references/riskLevel/riskReason/evidenceSpans/actions`。
    3.  **动作意图输出**：`add_clause` 仅输出意图（如 `targetParentId/supplementDraft`），不直接新增节点。
    4.  **策略过滤**：执行动作互斥策略；若判定 `delete`，则屏蔽 `revise` 与 `add_clause`。
*   **Stage C — 动作执行器 (Execution Pass, Deterministic)**：
    1.  **输入**：`nodes_stage_a` + `nodes_stage_b_actions`。
    2.  **执行**：按规则落地 `revise/delete/add_clause`，其中 `add_clause` 由程序生成新节点 ID、挂载 `parentId`、补边与排序。
    3.  **输出**：`nodes_final` + `patch_log`（可回放、可审计）。
    4.  **冲突处理**：同节点多动作冲突时按策略降级（如 `delete > revise > add_clause`）。
    5.  **高风险人工门控**：低置信度且高风险动作标记为“需人工复核”。

---

## 3. 核心 Prompt 设计（分阶段）

为保证结构稳定与可追溯性，Prompt 拆分为 Stage A / Stage B 两套指令，Stage C 由程序执行器实现（非 LLM 自由生成）。

### 3.1 Stage A Prompt（仅结构抽取）

这是直接给到 claude 3.5 的 Stage A 指令集：

```markdown
# Role
你是一个资深法律科技专家与合同结构化解析器。你的任务是将输入的非结构化合同文本转化为结构稳定的 JSON 节点数据（仅结构字段）。

# Core Constraints (不可违背)
1. **LOSSLESS CONTENT**: `content` 字段必须是原文的 100% 逐字复制。严禁概括、精简或修正原文中的任何字符（包括错别字）。
2. **HIERARCHICAL INTEGRITY (DETAILED RULES)**: 必须按以下“优先级 + 回退”规则确定 `parentId`，并覆盖复杂格式与条件条款：
   - **Rule A / 显式编号优先**：先识别编号体系并分组（如 `第一章/第一条/第(一)款`、`1/1.1/1.1.1`、`（一）/(1)/a)`、`A./B.`、`Section 1/1.1`）。
   - **Rule B / 同体系父子推断**：同一编号体系内，当前编号的直接上一级作为父节点（如 `1.2.3 -> 1.2`，`第(三)款 -> 第二条`）。
   - **Rule C / 编号跳级修复**：若出现跳级（如 `1.2.3` 之前无 `1.2`），挂到最近可用祖先。
   - **Rule D / 无编号段落挂载**：无编号但紧随某条款后的说明/定义/备注，默认挂到最近上文编号节点；若出现“定义如下/包括但不限于”等引导语，可作为该节点的 `sub`。
   - **Rule E / 条件条款拆分**：含 `若/如果/除非/在...情况下/provided that/subject to/unless/if` 的条件语句，需拆为子节点或至少打条件标签；其 `parentId` 指向触发该条件的主义务条款。
   - **Rule F / 列表项处理**：冒号后列表（`(1)(2)(3)`、`a,b,c`、`①②③`）视为同父级并列 `sub`，父节点为引导句所在条款。
   - **Rule G / 跨页与标题噪声**：页眉页脚、目录、签章页不参与层级；跨页断句先拼接再判断层级。
   - **Rule H / 冲突消解顺序**：`显式编号 > 排版缩进/标点线索 > 关键词线索 > 邻近位置`。若仍冲突，选择更保守父节点。
   - **Rule I / 可追溯性**：保持原始段落顺序，不得打乱节点出现顺序。
3. **STRUCTURE-ONLY OUTPUT**: 只允许输出 `id,label,content,type,parentId,timePhase`，禁止输出 `references/risk/actions`。
4. **FIELD ALIGNMENT**: 所有输出字段必须严格匹配下方定义的枚举值。

# Value Definitions
- **type (NodeKind)**:
  - `main`: 合同的核心章节或独立条款（如：付款方式、保密义务）。
  - `sub`: 条款下的具体细则、补充说明、参数或例外情况（挂在对应的 main 节点下，支持嵌套）。
- **timePhase**:
  - `pre_sign`: 签约前准备
  - `effective`: 合同生效/背景
  - `execution`: 合同执行/履行
  - `acceptance`: 验收/交付
  - `termination`: 合同终止/解约
  - `post_termination`: 终止后义务/保修

# Task Steps
1. **Pre-scan**: 扫描全文，识别所有条款边界、编号体系、标题与列表块。
2. **Hierarchy Normalization**: 按 Core Constraints 中 Rule A-I 构建层级树（root -> main -> sub）。
3. **Condition Segmentation**: 对条件/例外/触发条款进行拆分，确保条件片段有明确 `parentId`。
4. **TimePhase Mapping**: 根据条款内容推断时间阶段。
5. **JSON Synthesis**: 按下方 Schema 输出结构层结果。

# Output Schema (Strict JSON)
{
  "nodes": [
    {
      "id": "string (使用条款编号，如 'clause_1_1')",
      "label": "string (4-6字标题)",
      "content": "string (原文逐字复制)",
      "type": "main | sub",
      "parentId": "string (父节点ID，根节点指向 'root')",
      "timePhase": "pre_sign | effective | execution | acceptance | termination | post_termination"
    }
  ]
}

# Input Contract Text
[在此处输入合同全文]
```

### 3.2 Stage B Prompt（推理 + 动作意图）

Stage B 输入为 `nodes_stage_a` + `original_contract_text`，禁止修改 Stage A 字段，专注补全风险、引用与建议。以下是增强版 Prompt（工程可执行）：

```markdown
# Role
你是合同风险与关系推理引擎（Stage B）。
你的任务是基于既定结构节点，补全风险、引用关系与修改建议意图（不直接改结构）。

# Objective
基于 `nodes_stage_a` 与 `original_contract_text`，为每个节点补全：
- `references`
- `riskLevel`
- `riskReason`
- `actions`（delete/revise/add_clause 的意图）

# Non-Negotiable Constraints
1. **IMMUTABLE STRUCTURE**:
   不得修改 `id,label,content,type,parentId,timePhase`。
2. **EVIDENCE GROUNDED**:
   所有 `riskLevel != none` 的节点必须提供 evidence；证据文本必须能在该节点 `content` 中匹配。
3. **NO-RISK POLICY**:
   若 `riskLevel == none`，则 `actions` 必须为空数组。
4. **ACTION EXCLUSIVITY**:
   若某节点存在 `delete` 动作，不得同时出现 `revise` 或 `add_clause`。
5. **REFERENCE VALIDITY**:
   `references` 只能引用输入节点集中存在的 id。
6. **CONSERVATIVE INFERENCE**:
   证据不足时降低风险等级并给出保守结论，不得编造事实。
7. **INTENT-ONLY FOR ADD**:
   `add_clause` 只能输出新增意图（目标父节点与草案），不得在 Stage B 直接新增节点。

# Reasoning Pipeline (必须按顺序执行)
1. **Reference Extraction**
   - 识别显式引用（如“见第X条”“前款”“附件”）与高置信隐式依赖。
   - 去噪：每节点最多 5 条 references，仅保留最相关项。
2. **Risk Estimation**
   - 输出 `riskLevel` 与 `riskReason`。
   - 重点检查：责任失衡、期限模糊、违约后果缺失、触发条件不清、定义冲突、引用断裂。
3. **Action Suggestion (Intent)**
   - `delete`：仅用于明显冲突/重复/非法或不可修复矛盾文本。
   - `revise`：可通过改写澄清的风险，必须提供完整替换文案 `suggestionText`。
   - `add_clause`：关键保护缺失时补充，必须提供 `targetParentId` 与 `supplementDraft`。
   - 每节点最多 2 条建议，避免建议过载。
4. **Consistency Check**
   - 校验风险与建议一致性；校验证据与理由对齐。

# Confidence Rules
- 证据明确、规则清晰 -> 高
- 需上下文推断、证据弱 -> 中/低

# Output Schema (Strict JSON)
{
  "nodes": [
    {
      "id": "string (与输入一致)",
      "references": ["string"],
      "riskLevel": "none | low | medium | high",
      "riskReason": "string",
      "evidenceSpans": [{ "text": "string" }],
      "actions": [
        {
          "id": "string",
          "type": "delete | revise | add_clause",
          "status": "pending",
          "reason": "string",
          "suggestionText": "string",
          "supplementDraft": "string",
          "targetParentId": "string (仅 add_clause 必填)",
          "confidence": number
        }
      ]
    }
  ]
}

# Output Requirements
- 仅输出 JSON，不输出解释性文字或 markdown。
- 节点顺序保持与输入一致。
- 不得输出输入中不存在的 id。
- 空字段请用空字符串或空数组，不省略关键键名。
```

### 3.3 Stage C 执行器规范（程序侧，非 LLM）

Stage C 不再做语义推理，只执行 Stage B 的动作意图，确保结构变更可控、可回放。

**输入：**
- `nodes_stage_a`（结构骨架）
- `nodes_stage_b`（风险 + 动作意图）

**执行规则：**
1. `revise`：替换目标节点 `content`。
2. `delete`：删除目标节点（或标记删除），并处理关联边。
3. `add_clause`：
   - 必须存在 `targetParentId`。
   - 程序生成新节点 `id`，`parentId=targetParentId`，`type=sub`。
   - 新节点内容来自 `supplementDraft`，并建立父子边。
4. 冲突优先级：`delete > revise > add_clause`。

**输出：**
- `nodes_final`
- `patch_log`（记录动作、时间、操作者、前后差异）

---

## 4. 防御机制与后处理 (Post-Processing, Three-Pass)

由于 LLM 仍有幻觉概率，系统需在三阶段结果后执行自动化校验：

1.  **Stage A 字符级完整性校验**：
    `if (all_nodes_content.length < original_text.length * 0.98) { trigger_retry(); }`
2.  **Stage A 层级一致性校验**：
    `all node.parentId must exist OR be 'root'`
3.  **Stage B 字段不可变校验**：
    `assert stage_b[id].content/type/parentId/timePhase/label == stage_a[id].content/type/parentId/timePhase/label`
4.  **Stage B 证据坐标自动映射**：
    程序自动运行 `node.content.indexOf(evidence_text)` 来计算前端所需的 `start` 和 `end` 偏移量。
5.  **Stage B 引用合法性过滤**：
    `node.references = node.references.filter(refId => all_node_ids.includes(refId));`
6.  **Stage C 执行合法性校验**：
    `add_clause.targetParentId` 必须存在；新增节点 `id` 不得冲突；删除动作不得破坏根节点。
7.  **Stage C Patch 审计校验**：
    每条动作必须可在 `patch_log` 回放，且能重建 `nodes_final`。

## 5. 数据契约 (Data Contract)

上游最终输出的 JSON 结构示例：

```json
{
  "nodes": [
    {
      "id": "clause_5_2",
      "label": "违约责任",
      "content": "如乙方逾期交付，每逾期一日应支付合同总价 0.1% 的违约金...",
      "type": "sub",
      "parentId": "clause_5",
      "references": ["clause_3_1"],
      "riskLevel": "medium",
      "riskReason": "违约金比例偏低，可能无法覆盖实际损失。",
      "evidenceSpans": [
        { "text": "0.1% 的违约金" }
      ],
      "timePhase": "execution",
      "actions": [
        {
          "id": "act_001",
          "type": "revise",
          "status": "pending",
          "reason": "将违约金比例提升至行业标准 0.5%",
          "suggestionText": "如乙方逾期交付，每逾期一日应支付合同总价 0.5% 的违约金...",
          "confidence": 0.85
        }
      ]
    }
  ]
}
```
