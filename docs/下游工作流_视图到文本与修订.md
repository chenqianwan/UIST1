# 合同星座 (Contract Constellation) 下游工作流设计方案

本方案描述了如何将「合同星座」前端视图中的节点操作，稳定地转换为可审计的合同修订文本。设计遵循 **“先确定性落地，再模型润色”** 原则，强调 **可回放、可追踪、可回滚**。

---

## 1. 设计原则 (Design Principles)

1. **Event-First**：每次用户或 AI 动作都记录为结构化事件，不丢失交互历史。
2. **Tree-Diff as Source of Truth**：最终提交前以 `base_tree` vs `current_tree` 的 tree diff 作为“净变更集”。
3. **Deterministic Before LLM**：先程序化应用变更得到 `draft_v1`，再交给大模型做受约束整理。
4. **Incremental Diff First**：默认走增量 diff（仅扫描已操作节点），并周期性执行全量 diff 校验一致性。

---

## 2. 核心工作流：两阶段下游流水线 (Two-Pass Downstream Pipeline)

* **任务**：从视图到文本采用“两步走”：
  1) `tree diff + 规则引擎`（纯程序）生成可执行草稿；
  2) `thinking 模型` 仅做“严格映射应用”，不做额外改写。

### Stage D — 结构化变更编译 (Diff Compile Pass)

> Stage D 为 **程序执行阶段**，不调用大模型。

1. **输入**
   - `original_contract_text`
   - `base_tree`（初始结构，通常来自 Stage A）
   - `current_tree`（当前画布结构）
2. **Tree Diff 生成**
   - 优先读取节点 `touched/subtreeDirty` 标记，构建候选集合（增量 diff）。
   - 仅在候选集合内产出 `add / delete / revise` 三类操作。
   - 按树层级做祖先-后代冲突检查。
3. **冲突归并 (Normalization)**
   - `ancestor delete` 覆盖后代 `revise/add`
   - `add` 后立即 `delete` 的节点可抵消
4. **确定性应用**
   - 将 `normalized_diff` 作用到基线文本/结构，生成 `draft_v1`
   - 同步生成 `compile_report`（每条 diff 的执行状态）
5. **输出**
   - `normalized_diff`
   - `draft_v1`
   - `compile_report`

### Stage E — 严格映射应用 (Strict Mapping Pass)

1. **输入**
   - `original_contract_text`
   - `draft_v1`
   - `normalized_diff`
   - `compile_report`
2. **允许模型执行的任务**
   - 仅将 `draft_v1` 中已确定改动映射回 `original_contract_text`
   - 仅处理与映射相关的最小格式修正（如必要换行对齐）
3. **禁止模型执行的任务**
   - 执行新的 `add/delete/revise` 业务动作
   - 对未触达条款进行任何内容改写
   - 在触达条款上进行超出 `draft_v1` 的自由润色
   - 新增未授权义务或责任
   - 修改未出现在 diff 中的实质条款含义
   - 删除未授权节点
4. **输出**
   - `final_text`
   - `change_report[]`（每条文本改动绑定 `opId`）

## 3. 下游数据契约 (Data Contract)

### 3.1 Event Log（操作轨迹）

```json
{
  "eventId": "evt_20260303_001",
  "ts": "2026-03-03T12:34:56Z",
  "operator": "user",
  "nodeId": "article_3_1",
  "actionType": "revise",
  "before": "原条款文本...",
  "after": "修订后条款文本...",
  "reason": "期限定义过于模糊",
  "source": "manual",
  "applied": true,
  "touchVersion": 12
}
```

### 3.1.1 节点增量标记（用于高性能 Tree Diff）

```json
{
  "nodeId": "article_3_1",
  "touched": true,
  "subtreeDirty": false,
  "touchVersion": 12,
  "lastOpType": "revise",
  "deletedAtVersion": null
}
```

说明：
- `touched`：该节点是否被直接操作过。
- `subtreeDirty`：该节点子树是否受父级操作影响（如祖先 `delete`）。
- `touchVersion`：最近一次变更版本号，用于增量扫描与周期校验一致性。
- `deletedAtVersion`：软删除墓碑版本，避免已删节点被重复纳入 diff。

### 3.2 Tree Diff（净变更）

```json
{
  "meta": {
    "baseVersion": "v1",
    "currentVersion": "v9",
    "generatedAt": "2026-03-03T12:40:00Z"
  },
  "ops": [
    { "opId": "op_001", "type": "delete", "nodeId": "article_7", "subtree": true },
    {
      "opId": "op_002",
      "type": "revise",
      "nodeId": "article_3_1",
      "before": "原文...",
      "after": "修订文..."
    },
    {
      "opId": "op_003",
      "type": "add",
      "nodeId": "new_abc",
      "parentId": "article_5",
      "position": 2,
      "label": "5.3",
      "content": "新增条款..."

    {
      "opId": "op_099",

}
```

### 3.3 Finalize 输出

```json
{
  "final_text": "......",
  "change_report": [
    {
      "opId": "op_002",
      "changeType": "revise",
      "summary": "将租金支付期限由模糊表述改为明确工作日",

  ]
}
```

## 4. 冲突归并规则 (Normalization Rules)

1. **祖先删除优先**：若节点被祖先 `delete` 覆盖，则该节点所有操作转入 `suppressedOps`。
2. **同节点最终写入优先**：同一节点多次 `revise`，仅保留最后一次有效版本。
3. **新增后删除抵消**：同会话内 `add` 后 `delete` 且未被引用，可从最终 diff 中移除。
4. **输出拓扑顺序**：父节点操作先于子节点操作，避免应用顺序错误。
5. **增量标记传播**：父节点 `delete` 时必须将后代标记为 `subtreeDirty=true`。
6. **周期全量校验**：增量结果需与定期全量 diff 对齐，不一致时以全量结果为准并修正标记。

## 5. Stage E Prompt 模板（建议）

```markdown
# Role
你是合同文本一致性修订引擎（Stage E）。

# Objective
基于 `draft_v1` 与 `normalized_diff` 输出最终合同文本 `final_text`，
并生成可审计的 `change_report`。

# Hard Constraints
1. 只允许依据 `normalized_diff` 做改动。
2. 不得新增未授权义务、责任或付款条件。
3. 不得修改未触达条款的实质语义。
4. 所有改动必须在 `change_report` 中绑定 `opId`。

# Allowed Improvements
- 章节/条款编号连续性
- 交叉引用修复
- 术语统一（定义名词一致）
- 法律文体与语句顺滑

# Output Schema (Strict JSON)
{
  "final_text": "string",
  "change_report": [
    {
      "opId": "string",
      "changeType": "add|delete|revise|format_fix|reference_fix|term_unify",
      "summary": "string",
      "affectedSections": ["string"]
    }
  ]
}
```

---

## 6. 接口建议 (API Sketch)

1. `POST /downstream/events`：写入单条操作事件。
2. `POST /downstream/diff`：基于 `base_tree/current_tree` 生成 `normalized_diff`。
3. `POST /downstream/compile`：将 `normalized_diff` 确定性应用，产出 `draft_v1`。
4. `POST /downstream/finalize`：调用 thinking 模型，产出 `final_text + change_report`。

---

## 7. 验收标准 (Acceptance Criteria)

1. **可追溯**：`final_text` 每条关键改动都可回溯到 `opId`。
2. **可复现**：同一输入重复执行得到一致 `draft_v1`（确定性）。
3. **可控性**：模型输出无越权新增义务。
4. **可回滚**：删除某 `opId` 后可重新编译并得到可预期文本。
5. **可用性**：用户可在 UI 中查看“原文 -> 草稿 -> 终稿”的差异链路。
