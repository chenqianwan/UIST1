# 合同星座工作流总览

本文用一份文档说明我们当前的端到端工作流：从原始合同输入，到可视化分析，再到可审计的修订输出。

## 1. 总体目标

- 把非结构化合同文本转成可交互的条款图谱（Contract Constellation）。
- 让风险识别、修订建议、人工干预形成闭环。
- 最终输出可追溯、可回滚、可审计的合同修订结果。

---

## 2. 按功能块的系统说明（技术 -> 解决的问题 -> 效果）

### 功能块 A：合同结构化解析

- 用了什么技术
  - `LLM + 结构化 Schema 约束`（严格字段与枚举）。
  - `规则优先的层级锚定`（编号体系、父子关系、列表与条件条款拆分）。
  - `无损抽取约束`（content 必须逐字保留）。
- 解决了什么问题
  - 法律文本天然是长文且嵌套深，人工拆结构成本高且一致性差。
  - 直接做端到端“自由生成”容易出现幻觉、漏条款、层级错位。
- 做出的效果
  - 得到稳定的“条款树”作为后续风险分析与修订的共同底座。
  - 结构可复核、可对齐原文，降低法律语义丢失风险。

### 功能块 B：风险识别与建议生成

- 用了什么技术
  - `基于结构节点的风险推理`（none/low/medium/high）。
  - `引用关系抽取`（references）与 `动作策略约束`（delete/revise/add_clause 互斥）。
  - `保守推理策略`（证据不足时不强行判高风险）。
- 解决了什么问题
  - 法律审阅中“发现问题”和“提出可执行修改”常被割裂。
  - 纯分类模型难直接驱动实际修订动作。
- 做出的效果
  - 风险结果可直接转成可执行动作，减少从“判断”到“落地”的断层。
  - 对高风险节点优先暴露，提升审阅效率与可解释性。

### 功能块 C：可视化交互与人机协作

- 用了什么技术
  - `React + TypeScript` 构建状态驱动交互。
  - `SVG + 力导向布局`（`requestAnimationFrame`）展示条款关系网络。
  - `BFS 焦点扩散` + 动效（`Framer Motion`）强化可读性和路径感。
- 解决了什么问题
  - 线性阅读难以把握跨条款依赖与风险传播。
  - 全自动改写不符合法律场景的责任边界与可控性要求。
- 做出的效果
  - 用户可在图上快速定位核心条款、上下位条款与关联节点。
  - 形成“AI 提议 + 人工确认”的交互闭环，保证人在回路中。

### 功能块 D：变更编译与冲突归并

- 用了什么技术
  - `tree diff`（add/delete/revise）与 `normalization` 规则引擎。
  - 冲突处理（祖先删除覆盖后代修改、同节点最终写入优先）。
  - `deterministic compile`（先程序化生成 `draft_v1`）。
- 解决了什么问题
  - 多次编辑后，变更集合常出现重复、冲突、顺序错误。
  - 直接让模型“看全局改全文”很难审计与回滚。
- 做出的效果
  - 变更可压缩为净操作集，执行路径稳定、可重放。
  - 支持按 `opId` 追踪每一条变更的来源和影响。

### 功能块 E：受约束的终稿生成与审计

- 用了什么技术
  - `受约束 LLM 映射`（仅映射已批准变更，不越权新增义务）。
  - `change_report` 绑定 `opId` 的审计报告机制。
  - “最小必要格式修正”策略，控制模型自由度。
- 解决了什么问题
  - 法律文本最终交付既要求可读，也要求责任可追溯。
  - 传统“黑箱式润色”难满足合规审计需求。
- 做出的效果
  - 终稿文本与变更证据链一一对应，支持审计、复盘与回滚。
  - 在可控前提下提升文本质量，降低越权改写风险。

## 3. 给法律同学的 related work 检索地图

这部分用于论文写作：按功能块找相关研究，而不是按工程数据流找。

### A. 结构化解析（Legal Document Structuring）

- 建议检索词（中英混搜）
  - `legal contract structure extraction`
  - `hierarchical clause parsing`
  - `法律合同 条款层级 抽取`
  - `schema-constrained information extraction legal`
- 重点关注的问题类型
  - 条款边界识别、编号体系归一化、跨页/噪声处理、lossless extraction。
- 可对标的论文类型
  - 法律 NLP 信息抽取、文档结构解析、长文档分段建模。

### B. 风险识别与建议（Legal Risk + Actionability）

- 建议检索词
  - `contract risk assessment NLP`
  - `actionable legal recommendation`
  - `法律条款 风险识别 可执行建议`
  - `human-in-the-loop legal AI review`
- 重点关注的问题类型
  - 风险等级定义、证据可解释性、建议可执行性、保守推理策略。
- 可对标的论文类型
  - 法律风险预测、法律文本决策支持、可解释 AI in law。

### C. 可视分析与交互（Visual Analytics for Legal Text）

- 建议检索词
  - `legal text visualization`
  - `clause dependency graph`
  - `interactive legal document analysis`
  - `法律文本 可视化 交互 分析`
- 重点关注的问题类型
  - 从线性文本到网络图、认知负担降低、专家用户任务效率。
- 可对标的论文类型
  - HCI/InfoVis/VA（CHI, UIST, VIS）中的专业文本可视分析系统。

### D. 变更管理与可审计生成（Diff + Traceability）

- 建议检索词
  - `legal document revision tracking`
  - `traceable AI text editing`
  - `policy-constrained text generation`
  - `可审计 文本生成 法律`
- 重点关注的问题类型
  - 变更溯源、版本控制、冲突归并、可回滚机制。
- 可对标的论文类型
  - 文档工程、可追踪生成系统、合规场景的人机协作写作。

### E. 论文写作时可用的叙事框架

- 我们的定位不是“替代律师”，而是“增强法律审阅流程的可视-可控-可审计基础设施”。
- 技术贡献可以表述为三层：
  - 结构层：稳定、无损、可复核的合同结构化。
  - 交互层：风险可视化与人在回路中的修订决策。
  - 治理层：可追溯、可回滚、可审计的终稿生成机制。

---

## 4. 最终交付物

- 结构与语义：`nodes_stage_a / nodes_enriched`
- 交互与过程：`current_tree / event_log / normalized_diff / compile_report`
- 结果与审计：`draft_v1 / final_text / change_report`

以上三类产物共同支持论文中的核心主张：系统不仅“能生成”，而且“能解释、能控制、能审计”。
